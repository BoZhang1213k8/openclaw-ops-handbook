# 19. 群聊多人并发操作的确认安全

## 19.1 问题背景

在飞书或 Telegram 群聊中，机器人与所有群成员共享同一个对话会话。当多人同时操作不同的 Skill 时，会产生一个关键问题：

> **用户 A 执行 Skill 1（如删除记录），Bot 提示"请确认是否删除？"**
> **此时用户 B 正在执行 Skill 2（如 PBOSS 回单），回复了"确认"**
> **这个"确认"会不会误触发用户 A 的删除操作？**

**答案是：存在误触发的风险。**

这不是理论上的担忧，而是在群聊多人协作场景中真实会遇到的问题，尤其是涉及删除、重置等不可逆操作时，后果可能很严重。

## 19.2 根因分析

### 原因一：会话按群共享，不区分用户

OpenClaw 的会话模型是 **每个群一个会话**：

- 会话 ID 格式：`feishu:group:<group_id>` 或 `telegram:group:<group_id>`
- 同一群内所有用户的消息进入 **同一个对话上下文**
- LLM 看到的是一连串消息流，无法可靠区分"这条消息是谁发的"

```
会话上下文（同一个群）
┌──────────────────────────────────┐
│ [用户A] 删除 MBH SN 123         │
│ [Bot]   SN 123 有1条记录，确认？  │
│ [用户B] 重发 PBOSS 回单 139xxx   │
│ [Bot]   查到2条记录，确认？       │
│ [???]   确认                     │  ← LLM：这是谁在确认什么？
└──────────────────────────────────┘
```

### 原因二：确认机制是纯提示词约束，无编程级校验

所有 Skill 的确认流程都通过 SKILL.md 中的提示词实现：

```
步骤2：预览查询 → 展示结果
步骤3：询问"请确认是否执行？" → 停止回复，等待用户输入
步骤4：用户确认后执行
```

这个"等待确认"是 **LLM 的行为约束**，不是程序锁：

- LLM 无法验证"回复确认的人"是否就是"发起操作的人"
- LLM 无法区分"这个确认是针对哪个待确认操作的"
- 没有操作 ID、没有用户 ID 绑定、没有超时机制

### 原因三：串行处理的危险时序

Gateway 对同一群的请求串行处理。看似安全，实际加剧了问题：

```
时序  消息                                会话状态
────  ────────────────────────────────  ──────────────────────────
T1    用户A @Bot "删除 MBH SN 123"      → Bot 查询
      Bot: "SN 123 有1条记录，确认删除？"   【待确认操作 1：删除】

T2    用户B @Bot "重发 PBOSS 139xxx"    → Bot 查询
      Bot: "查到2条记录，确认回单哪条？"    【待确认操作 1+2 并存】

T3    用户B 回复 "确认"                  → LLM 看到最近的是 PBOSS 回单
                                          大概率执行 PBOSS 回单 ✅
                                          ⚠️ 但也可能误触发删除操作

T4    用户A 回复 "确认"                  → 上下文已被 T2/T3 冲淡
                                          ⚠️ LLM 可能找不到之前的删除操作
                                          ⚠️ 可能关联到错误的操作
```

## 19.3 风险等级评估

| 场景 | 风险 | 说明 |
|------|:---:|------|
| 不同群的用户分别操作 | ✅ 无风险 | 会话完全隔离，互不影响 |
| 同群只有一人操作 | ✅ 无风险 | 不存在确认混淆的可能 |
| 同群两人先后操作，间隔较长（>5分钟） | ⚠️ 低风险 | 第一人操作已完成，上下文清晰 |
| 同群两人操作时间重叠（都在等确认） | 🔴 高风险 | 确认消息可能被 LLM 关联到错误的操作 |
| 同群多人同时操作同类 Skill | 🔴🔴 极高风险 | 操作类型相同，LLM 几乎无法区分 |

**最危险的组合**：两人在同一群内同时执行需要确认的不可逆操作（如删除 + 归档），且时间重叠。

## 19.4 已有的基础缓解机制

以下机制对此问题有一定帮助，但均 **不能根本解决**：

| 机制 | 配置 | 对此问题的效果 |
|------|------|:---:|
| ACK 反应 | `ackReactionScope: "group-mentions"` | ⚠️ 仅标记"已收到"，不解决确认归属 |
| 块式流回复 | `blockStreaming: true` | ⚠️ 回复不交织，但不解决确认归属 |
| 必须 @触发 | `requireMention: true` | ⚠️ 初始指令需 @，但确认回复通常不带 @ |
| 串行处理 | Gateway 内部机制 | ⚠️ 保证不并行执行，但不能防止确认错配 |

## 19.5 解决方案：特定确认短语机制（已实施）

### 核心思路

将所有 Skill 的确认从通用的"请确认"改为要求 **特定确认短语**，短语中必须包含 **操作关键词 + 业务参数**。LLM 被指示拒绝不含关键词的通用确认。

### 改动前后对比

**改动前**（危险）：

```
Bot: SN 123 存在 1 条记录，确认后可删除。请确认是否执行删除？
用户: 确认        ← 任何人的"确认"都可能触发
```

**改动后**（安全）：

```
Bot: SN 123 存在 1 条记录。请回复"确认删除 SN 123"执行删除。
用户: 确认删除 SN 123    ← 包含操作类型和具体参数，不易被混淆
```

### 各 Skill 的确认短语

| Skill | 确认短语格式 | 示例 |
|-------|-------------|------|
| mbhopen-delete | "确认删除 SN xxx" | "确认删除 SN A1B2C3" |
| pboss-resend | "确认回单 记录ID" | "确认回单 100235,100236" |
| ticket-restart | "确认重启 记录ID" | "确认重启 752321" |
| order-archive | "确认归档 工单ID" | "确认归档 123456" |
| file-resend | "确认补传 文件类型" | "确认补传 大音20指标" |

### 为什么有效

1. **操作关键词唯一** — 删除/回单/重启/归档/补传，每个 Skill 关键词不同
2. **业务参数绑定** — SN/记录ID/工单ID/文件类型，进一步限定了操作对象
3. **拒绝通用确认** — LLM 被指示对只回复"确认"、"好的"、"可以"的消息要求补充完整短语
4. **即使并发也能区分** — 当群内同时有"确认删除"和"确认回单"两个待确认操作时，"确认删除 SN 123"能明确关联到删除操作

**并发场景示例**：

```
时序  消息                                  LLM 判断
────  ────────────────────────────────────  ──────────────────────────
T1    用户A: 删除 MBH SN 123
      Bot: "请回复'确认删除 SN 123'执行"     【待确认：删除 SN 123】

T2    用户B: 重发 PBOSS 回单 139xxx
      Bot: "请回复'确认回单 100235'执行"     【待确认：删除 + 回单】

T3    用户B: "确认回单 100235"              → 匹配"回单"+记录ID → 执行回单 ✅

T4    用户A: "确认删除 SN 123"              → 匹配"删除"+SN → 执行删除 ✅

T?    某人: "确认"                          → 不含操作关键词 → 拒绝，要求补充 ✅
```

### 涉及的配置文件

此方案已同步更新到以下文件：

| 文件 | 修改内容 |
|------|---------|
| 各 Skill 的 `SKILL.md`（步骤3 + 步骤4） | 确认提示改为特定短语，新增拒绝通用确认的规则 |
| `AGENTS.md`（Skill 执行规范 第3条） | "确认先行"改为"特定确认短语机制"，说明设计目的 |
| `TOOLS.md`（各 Skill 的执行步骤和重要规则） | 同步更新确认短语要求 |

### 局限性

此方案是 **提示词级别** 的改进，不是编程级别的强校验：

- LLM 可能在极端情况下仍然忽略约束
- 用户可能不按要求回复完整短语，导致多次交互
- 无法验证"回复确认的人"是否就是"发起操作的人"（只能通过短语内容间接关联）

但在当前 OpenClaw 的架构下，这是 **最有效的可行方案**，能覆盖绝大多数场景。

## 19.6 操作规范建议

除了技术层面的改进，团队在日常使用中也应遵循以下规范：

### 规范一：避免并发操作

**同一群内同一时间只有一人操作需要确认的 Skill。** 一人操作完成后，下一人再发起。

这是最简单、最有效的规避方式。如果团队人数多、操作频繁，考虑使用私聊（见规范三）。

### 规范二：严格按提示回复

看到 Bot 提示"请回复'确认删除 SN xxx'"时，**严格按照提示的格式回复**，不要简化为"确认"或"好的"。

### 规范三：高危操作走私聊

对于删除、重置等不可逆操作，建议通过飞书私聊执行，彻底避免确认混淆：

```jsonc
// openclaw.json
"channels": {
  "feishu": {
    "dmPolicy": "allowlist",
    "allowFrom": ["ou_xxxxxxxxxxxx"]   // 管理员的飞书 open_id
  }
}
```

私聊是完全独立的会话（`feishu:dm:<user_id>`），不与群聊共享上下文。

### 规范四：关注 ACK 反应

发送指令后，注意 Bot 是否给消息加了 emoji 反应（ACK）。如果没有反应，说明消息可能未被处理（可能在排队等待前一个操作完成）。

## 19.7 长期改进方向（需 OpenClaw 平台支持）

以下改进需要 OpenClaw 平台层面的支持，当前版本（2026.2.9）暂不可用：

| 改进方向 | 描述 | 效果 |
|---------|------|------|
| 按用户隔离群内子会话 | 为同一群内的不同用户维护独立的操作上下文 | 彻底解决确认混淆 |
| 编程级确认机制 | 引入飞书交互卡片/按钮，绑定用户身份和操作 ID | 确认与用户身份强绑定 |
| 确认超时机制 | 待确认操作超时未确认自动取消 | 防止陈旧的待确认操作被误触发 |
| 回复关联发起人 | Bot 回复时自动 @发起操作的用户 | 提高可读性，便于区分 |

## 19.8 适用范围

本章描述的问题和解决方案适用于所有 **群聊渠道**：

| 渠道 | 是否受影响 | 说明 |
|------|:---:|------|
| 飞书群聊 | ✅ 受影响 | 会话按群共享 |
| Telegram 群聊 | ✅ 受影响 | 会话按群共享 |
| 飞书私聊 | ❌ 不受影响 | 每人独立会话 |
| Telegram 私聊 | ❌ 不受影响 | 每人独立会话 |
| Web 聊天 / HTTP API | ❌ 不受影响 | 每个连接独立会话 |
